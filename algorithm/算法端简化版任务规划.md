# 算法端任务规划（简化版）

## 一、算法端定位

算法端主要负责本项目中的数据设计、模拟数据生成、异常检测和负载预测。根据当前项目需求，算法端不需要做复杂模型训练，也不需要做完整电器分类，只需要通过规则算法和模拟数据实现系统的“智能分析”效果。

本版本算法端重点包括：

1. 宿舍用电模拟数据生成
2. 无人宿舍耗电检测
3. 高功率违规用电检测
4. 楼层过载风险检测
5. 节能建议与预警信息生成



---

## 二、算法端整体目标

算法端的目标不是开发复杂的人工智能模型，而是为软件端提供可以展示的数据和判断结果，使系统具备以下能力：

1. 能显示每个宿舍的实时用电情况
2. 能判断宿舍是否存在异常用电
3. 能识别疑似高功率违规用电行为
4. 能判断楼层是否存在过载风险
5. 能预测晚间高峰时段的负载压力
6. 能向学生端和管理员端输出节能建议

因此，算法端可以采用：

> 模拟数据 + 规则判断 + 简单趋势预测

这种方式实现难度低，展示效果明显，适合期中项目原型。

## 三、算法端功能模块划分

## 模块一：模拟数据设计与生成

### 1. 主要任务

算法端首先需要生成一批宿舍能源数据，用于软件端页面展示和异常检测。

### 2. 建议模拟范围

可以模拟一栋宿舍楼，例如：

| 项目    | 建议设置      |
| ----- | --------- |
| 宿舍楼   | A 栋       |
| 楼层数   | 5 层       |
| 每层宿舍数 | 10 间      |
| 总宿舍数  | 50 间      |
| 数据时间  | 7 天       |
| 数据间隔  | 每 30 分钟一条 |

### 3. 数据字段设计

算法端需要保留的字段如下：

| 字段名              | 含义          | 示例                  |
| ---------------- | ----------- | ------------------- |
| dorm\_id         | 宿舍编号        | A101                |
| floor            | 楼层          | 1                   |
| time             | 时间          | 2026-05-19 18:30    |
| power            | 当前功率，单位 W   | 850                 |
| energy           | 累计能耗，单位 kWh | 5.6                 |
| temperature      | 温度          | 27.5                |
| occupied         | 是否有人        | true / false        |
| abnormal\_status | 是否异常        | normal / abnormal   |
| abnormal\_type   | 异常类型        | 无人耗电 / 高功率异常 / 楼层过载 |
| risk\_level      | 风险等级        | 低 / 中 / 高           |
| suggestion       | 系统建议        | 建议关闭无人宿舍用电设备        |

###

### 5. 模拟数据类型

数据中需要包含几类典型宿舍：

| 宿舍类型    | 数据特征                        | 用途            |
| ------- | --------------------------- | ------------- |
| 正常宿舍    | 功率较平稳，无异常                   | 展示正常用电        |
| 无人耗电宿舍  | occupied = false，但 power 较高 | 展示无人宿舍耗电预警    |
| 高功率异常宿舍 | power 突然超过阈值                | 展示疑似违规高功率设备预警 |
| 高耗能宿舍   | 长时间功率偏高                     | 展示能耗偏高提醒      |
| 楼层过载宿舍群 | 同一楼层多个宿舍功率较高                | 展示楼层过载风险      |

### 6. 输出成果

算法端需要向软件端提供：

```text
dorm_energy_data.json
```

该文件用于展示宿舍实时用电、能耗趋势和异常状态。

## 模块二：异常检测逻辑设计

### 1. 主要任务

异常检测模块负责判断宿舍用电是否存在风险。

本项目保留三类异常：

1. 无人宿舍耗电
2. 高功率异常用电
3. 楼层过载风险

不做夜间异常用电检测。

---

## 2. 无人宿舍耗电检测

### 判断逻辑

如果宿舍无人，但仍然存在较高功率用电，则认为存在能源浪费。

### 建议规则

```text
如果 occupied = false 且 power > 300W：
    判定为“无人宿舍耗电”
```

### 示例

| 宿舍   | 是否有人 | 当前功率 | 判断结果   |
| ---- | ---- | ---- | ------ |
| A101 | 有人   | 250W | 正常     |
| A102 | 无人   | 80W  | 正常     |
| A103 | 无人   | 520W | 无人宿舍耗电 |

### 输出结果

```json
{
  "dorm_id": "A103",
  "power": 520,
  "occupied": false,
  "abnormal_status": "abnormal",
  "abnormal_type": "无人宿舍耗电",
  "risk_level": "中",
  "suggestion": "检测到宿舍无人但仍有较高用电，建议关闭照明、插座或其他待机设备。"
}
```

---

## 3. 高功率异常用电检测

### 判断逻辑

本项目不再识别所有电器类型，只识别是否存在疑似高功率违规用电。

也就是说，不需要判断是电热锅、热得快还是其他设备，只需要判断：

> 当前功率是否明显超过普通宿舍设备的正常范围。

同时，空调不作为违规高功率设备处理。

### 建议规则

```text
如果 power > 1500W：
    判定为“疑似高功率违规用电”
```

### 空调排除说明

空调功率通常较高，但在宿舍场景中可能属于允许使用设备。因此本项目不把空调作为违规识别对象。

为了避免把空调误判为违规设备，可以设置一个简单说明：

```text
700W 至 1200W 的持续功率视为可能的正常空调用电，不判定为高功率违规。
超过 1500W 的突增或持续高功率，判定为疑似高功率违规用电。
```

### 示例

| 宿舍   | 当前功率  | 判断结果          |
| ---- | ----- | ------------- |
| A201 | 850W  | 正常，可能为空调或普通用电 |
| A202 | 1180W | 正常偏高，不作为违规    |
| A203 | 1750W | 疑似高功率违规用电     |
| A204 | 2100W | 高风险违规用电       |

### 风险等级设置

| 功率范围       | 风险等级 | 判断        |
| ---------- | ---- | --------- |
| 0–1200W    | 低    | 正常或普通高负载  |
| 1200–1500W | 中    | 用电偏高，需要关注 |
| 1500W 以上   | 高    | 疑似高功率违规用电 |

### 输出结果

```json
{
  "dorm_id": "A203",
  "power": 1750,
  "abnormal_status": "abnormal",
  "abnormal_type": "疑似高功率违规用电",
  "risk_level": "高",
  "suggestion": "检测到宿舍瞬时功率过高，疑似存在高功率设备使用，请管理员重点关注。"
}
```

---

## 4. 楼层过载风险检测

### 判断逻辑

楼层过载不是看单个宿舍，而是统计同一楼层所有宿舍的总功率。

### 建议规则

```text
计算每一层所有宿舍当前功率总和。
如果 floor_total_power > 8000W：
    判定为“楼层过载风险”
```

### 示例

| 楼层  | 当前总功率 | 判断结果   |
| --- | ----- | ------ |
| 1 楼 | 5200W | 正常     |
| 2 楼 | 7600W | 接近高峰   |
| 3 楼 | 8900W | 楼层过载风险 |
| 4 楼 | 6800W | 正常     |

### 风险等级设置

| 楼层总功率      | 风险等级 |
| ---------- | ---- |
| 0–6000W    | 低    |
| 6000–8000W | 中    |
| 8000W 以上   | 高    |

### 输出结果

```json
{
  "floor": 3,
  "floor_total_power": 8900,
  "abnormal_type": "楼层过载风险",
  "risk_level": "高",
  "suggestion": "当前楼层总负载较高，建议引导学生错峰使用高功率设备。"
}
```

## 模块三：高峰负载预测

### 1. 主要任务

高峰负载预测模块用于判断未来一段时间宿舍楼是否可能进入用电高峰。

该模块不需要复杂机器学习模型，可以采用时间规律和历史平均值进行简单预测。

### 2. 高峰时间设定

宿舍用电通常具有明显时间规律：

| 时间段         | 用电特征       |
| ----------- | ---------- |
| 07:00–09:00 | 起床时间，用电上升  |
| 12:00–14:00 | 中午回宿舍，用电中等 |
| 18:00–23:00 | 晚间高峰，用电最高  |
| 其他时间        | 用电相对较低     |

### 3. 预测规则

```text
如果当前时间接近 18:00–23:00，且楼层当前负载已经达到中等水平：
    预测未来可能出现高峰负载。

如果某楼层连续 3 个时间点总功率上升：
    预测未来 30 分钟负载继续上升。
```

### 4. 简单预测公式

可以使用最近三个时间点的平均增长量进行预测：

```text
预测负载 = 当前负载 + 最近三个时间点的平均增长量
```

例如：

```text
18:00 楼层总功率 = 6500W
18:30 楼层总功率 = 7100W
19:00 楼层总功率 = 7600W

平均增长量 = [(7100 - 6500) + (7600 - 7100)] / 2 = 550W
预测 19:30 负载 = 7600 + 550 = 8150W
```

此时预测值超过 8000W，可以判定为高峰负载风险。

### 5. 输出结果

```json
{
  "floor": 3,
  "current_load": 7600,
  "predicted_load": 8150,
  "risk_level": "高",
  "suggestion": "预计该楼层未来 30 分钟进入用电高峰，建议提前进行负载预警。"
}
```

## 十、汇报中可以这样介绍算法端

算法与数据分析端主要负责系统的智能分析能力建设，包括宿舍用电模拟数据生成、异常用电检测、楼层负载分析和高峰负载预测。系统通过采集和模拟宿舍功率、时间、是否有人、温度和楼层信息，对无人宿舍耗电、高功率异常用电以及楼层过载风险进行规则判断，并根据历史负载变化预测未来高峰用电情况。算法端最终向软件端输出实时用电数据、异常报警列表、楼层负载数据、预测结果和节能建议，从而支撑学生端与管理员端的可视化展示。
